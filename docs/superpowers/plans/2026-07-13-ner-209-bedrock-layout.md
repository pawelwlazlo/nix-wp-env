## Context

`nix-wp-env` is a reproducible single-site local WordPress dev environment built
on **devenv** (standard devenv â€” `devenv.yaml` + `devenv.nix`, **no top-level
`flake.nix`**). NER-208 landed the base stack on `main`: PHP 8.4 + PHP-FPM,
MariaDB (auto-creates a `wordpress` DB + `wordpress`/`wordpress` user), and Caddy
serving `web/` over plain HTTP on `:8080`, proxying `.php` to the PHP-FPM `web`
pool socket. Right now `web/` holds a **placeholder `index.php`** (phpinfo).

This issue replaces that placeholder with the Roots **Bedrock** layout so Caddy
serves a real WordPress docroot and Composer manages WP core + third-party
plugins. After this issue, hitting the site shows WordPress (the install wizard,
until NER-211 runs `wp core install`).

  and Composer-managed plugins are gitignored.
- **Caddy docroot stays `web/`** (already correct from NER-208); Bedrock's own
  `web/index.php` replaces the placeholder.

## Environment (how to actually run things on this machine â€” iMac-Pawe)

Neither `composer` nor `devenv` is installed globally. They come from the devenv
shell, which is itself run via nixpkgs' prebuilt devenv:

- **Get Composer + PHP 8.4:** enter the devenv shell â€”
  `nix run nixpkgs#devenv -- shell`. Inside it, `php` (8.4), `composer`, and
  `wp` are on PATH. Alternatively for a one-off Composer without the full shell:
  `nix shell nixpkgs#php84 nixpkgs#php84Packages.composer -c composer ...`.
- **Use `nix run nixpkgs#devenv`, NOT `github:cachix/devenv`.** The nixpkgs one
  is prebuilt (cache.nixos.org, ~2.1.2 on this host); the github flake builds
  from source (slow, no cache hit).
- **CRITICAL gitconfig gotcha:** the global `~/.gitconfig` has an `insteadOf`
  rule rewriting `https://github.com/` â†’ `git@github.com:`. devenv's embedded
  libnix (and nix flake fetching generally) honors it and fails with
  `authentication required but no callback set` (libgit2 has no SSH auth
  callback). `GIT_CONFIG_GLOBAL` does NOT fix it (libgit2 ignores that env).
  **Workaround:** run every `nix`/`devenv` command with `HOME` pointed at a temp
  dir containing an empty `.gitconfig`:
  ```bash
  export NIXHOME=$(mktemp -d); : > "$NIXHOME/.gitconfig"
  HOME="$NIXHOME" nix run nixpkgs#devenv -- shell -- <cmd>
  ```
  Normal `git` (commit/push) is unaffected â€” leave HOME alone for git.
- **`devenv up` flags:** this host's devenv (2.1.2) supports `up -d --no-tui`
  (detached). If a command errors with "unexpected argument --no-tui" you are on
  an older devenv â€” drop the flag.
- **MariaDB `wordpress` DB** is auto-created because `processes.caddy.after`
  depends on `devenv:mysql:configure` (already wired in NER-208). Don't remove
  that.
- **Network:** `composer create-project` / `composer install` and `nix run`
  need network (packagist, github, cache.nixos.org). If a command fails from
  sandbox network restrictions, retry it with the sandbox disabled
  (`dangerouslyDisableSandbox: true`) â€” these are trusted package fetches.

## Tasks

### Task 1 â€” Bedrock skeleton + Composer-managed WordPress

**Goal:** bring the Bedrock skeleton into the repo, adapt its env + gitignore to
this project, and populate WP core via Composer.

Steps:

1. Generate Bedrock into a temp dir:
   `composer create-project roots/bedrock <tmp>` (run via the devenv shell /
   nix so Composer + PHP 8.4 are available; use the HOME workaround for nix).
2. Copy the skeleton into the repo, resolving collisions with existing files
   (`devenv.*`, `docs/`, `README.md`, `.gitignore`, `web/index.php`):
   - `composer.json` (Bedrock deps: `roots/bedrock`, `roots/wordpress`,
     `roots/wp-config`, `vlucas/phpdotenv`, `oscarotero/env`,
     `roots/wordpress-core-installer`, `roots/bedrock-autoloader`, etc. â€” take
     whatever the generated skeleton produces; do not hand-invent versions).
   - `config/` (`application.php`, `environments/development.php`, etc.).
   - `web/index.php`, `web/wp-config.php` (Bedrock's â€” these replace/augment the
     NER-208 placeholder `web/index.php`).
   - `web/app/` structure: `themes/`, `plugins/`, `mu-plugins/`, `uploads/`.
     Keep the tracked dirs present via Bedrock's `.gitkeep`s (themes/plugins/
     mu-plugins). `uploads/` is gitignored.
   - `.env.example` â†’ adapt to the Global-Constraints values above.
3. Merge `.gitignore`: keep the existing NER-208 entries (`.idea/`, `.devenv/`,
   `.devenv.flake.nix`, `devenv.local.nix`, `.direnv/`) and add Bedrock/Composer
   ones: `.env`, `web/wp/`, `vendor/`, `web/app/uploads/`,
   `web/app/plugins/`, `web/app/mu-plugins/`. Do NOT ignore `web/app/themes/`.
   (Bedrock ships its own `.gitignore` â€” merge, don't blindly overwrite the
   NER-208 devenv entries.)
4. Run `composer install`. Confirm `web/wp/` is populated (WP core) and
   `vendor/` exists. This writes/updates `composer.lock` â€” commit it.
5. Commit the tracked deliverables: `composer.json`, `composer.lock`, `config/`,
   `web/index.php`, `web/wp-config.php`, tracked `web/app/**` `.gitkeep`s,
   `.env.example`, updated `.gitignore`. (Do NOT commit `web/wp/`, `vendor/`,
   `.env`.)

**Acceptance:**
- `composer.json` requires `roots/bedrock`; `composer.lock` is tracked.
- Bedrock structure present: `config/`, `web/` docroot, `web/wp/` (gitignored,
  populated), `web/app/{themes,plugins,mu-plugins,uploads}`.
- `.env.example` tracked with the exact dev defaults; `.env` absent + gitignored.
- `.gitignore` covers `.devenv/`, `.env`, `web/wp/`, `vendor/`,
  `web/app/uploads/`, Composer-managed plugins; `web/app/themes/` NOT ignored.
- `git status` shows no untracked `web/wp/` or `vendor/` (properly ignored).

### Task 2 â€” devenv wiring cleanup + end-to-end verification

**Goal:** ensure Caddy serves the Bedrock docroot and the site returns the
WordPress install screen; tidy the NER-208 "placeholder" wording in `devenv.nix`.

Steps:

1. In `devenv.nix`: the Caddy `root` is already `${config.devenv.root}/web`,
   which is correct for Bedrock â€” keep it. Update the NER-208 comments that call
   `web/` a "placeholder docroot" so they describe the Bedrock docroot. Confirm

   `web/index.php` in Task 1). Do NOT change the port, the FPM pool, the
   unix-socket syntax, or the `processes.caddy.after` DB-ordering.
2. Boot the stack (HOME workaround):
   `HOME=$NIXHOME nix run nixpkgs#devenv -- up -d --no-tui`.
   Wait for MariaDB + PHP-FPM + Caddy to come up.
3. Verify: `curl -sSL http://localhost:8080` returns the **WordPress install
   screen** (HTML from `/wp/wp-admin/install.php`, i.e. "WordPress â€ş
   Installation" / language-select), NOT phpinfo and NOT "Error establishing a
   database connection". A redirect to `/wp/wp-admin/install.php` is the
   expected success signal (DB reachable, WP not yet installed).
4. Tear down: `HOME=$NIXHOME nix run nixpkgs#devenv -- processes down` (or
   `devenv processes down`).

**Acceptance:**
- Caddy serves the Bedrock docroot (`web/`); NER-208 placeholder gone.
- `curl -L http://localhost:8080` reaches the WordPress install screen (DB
  reachable, not yet installed) â€” captured as evidence.
